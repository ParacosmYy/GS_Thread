# GS_Thread

`GS_Thread` 是一个类 RT-Thread 内核复刻与学习型 RTOS 项目，用于系统性理解线程调度、对象管理、IPC、软件定时器和 Cortex-M 移植机制。项目目标不是简单堆功能，而是把 RTOS 内核的关键路径拆成可阅读、可验证、可复盘的工程实现。

## 项目定位

| 项目 | 说明 |
| --- | --- |
| 类型 | RTOS 内核复刻 / 嵌入式系统学习项目 |
| 参考方向 | RT-Thread 内核机制、野火内核教程、Cortex-M 上下文切换 |
| 目标平台 | ARM Cortex-M 系列 MCU |
| 核心价值 | 理解调度器、线程模型、对象容器和移植层 |
| 分支职责 | 主线开发与结构化沉淀 |

## 核心模块

| 模块 | 说明 |
| --- | --- |
| Thread | 线程控制块、线程初始化、线程切换 |
| Scheduler | 就绪表、优先级调度、上下文切换入口 |
| Object | 对象容器、对象注册、统一管理模型 |
| Timer | 系统节拍、软件定时器、超时管理 |
| IPC | 信号量、互斥量、邮箱等同步通信机制 |
| Port | Cortex-M 汇编移植、PendSV、SysTick、临界区 |

## 推荐架构

```text
GS_Thread/
├── kernel/             # 调度器、线程、对象、IPC、定时器
├── port/               # Cortex-M 移植层与汇编上下文切换
├── bsp/                # 板级启动、时钟、串口、SysTick
├── examples/           # 内核功能验证用例
├── docs/               # 学习记录、设计说明、调试复盘
└── project/            # Keil / GCC 工程入口
```

## 学习路线

1. 先完成线程栈初始化和手动上下文切换。
2. 引入 PendSV，建立可控的线程切换路径。
3. 实现就绪表和调度器。
4. 实现对象容器，把线程、定时器、IPC 纳入统一管理。
5. 引入 SysTick 和软件定时器。
6. 补充 IPC，形成最小可用内核。
7. 整理测试用例和调试记录。

## 快速验证

```text
打开工程
    -> 编译内核
    -> 下载到开发板
    -> 观察串口输出或断点
    -> 验证线程切换顺序
```

关键验证点：

- 线程栈初始帧是否符合 Cortex-M 异常返回规则。
- PendSV 优先级是否低于业务中断。
- 临界区是否保护调度器和对象容器。
- SysTick 是否只负责节拍推进，不承载复杂业务。

## 质量标准

| 检查项 | 要求 |
| --- | --- |
| 可读性 | 内核关键路径有明确模块边界 |
| 可验证 | 每个内核能力至少有一个最小示例 |
| 可移植 | Cortex-M 相关代码集中在 `port` 层 |
| 可复盘 | 调试问题沉淀到文档，不只保留口头经验 |
| 可演进 | 新增 IPC 或对象类型不破坏对象管理模型 |

## 分支说明

| 分支 | 用途 |
| --- | --- |
| `main` | 当前主线开发与文档基线 |
| `master` | 历史兼容分支，建议与主线保持同步 |
| `release` | 阶段性稳定版本、学习记录和里程碑沉淀 |

## 维护边界

该项目以理解 RTOS 内核机制为主，不追求直接替代成熟 RTOS。复杂外设驱动、业务应用和平台适配应保持克制，避免干扰内核学习主线。
