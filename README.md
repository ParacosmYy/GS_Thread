# GS_Thread Master Branch

`master` 是 GS_Thread 的历史兼容分支，用于保留类 RT-Thread 内核复刻项目的稳定主线视图。后续建议逐步以 `main` 作为主要开发入口，`master` 仅保留兼容、同步和阶段性稳定用途。

## 分支定位

| 项目 | 说明 |
| --- | --- |
| 类型 | RTOS 内核复刻项目历史主线 |
| 参考方向 | RT-Thread 内核机制、Cortex-M 上下文切换 |
| 分支职责 | 稳定视图、兼容引用、阶段性同步 |
| 推荐上游 | `main` |
| 维护重点 | 与主线结构保持一致，避免分叉失控 |

## 项目范围

GS_Thread 聚焦 RTOS 内核关键机制：

- 线程控制块与线程栈初始化。
- PendSV / SysTick 驱动的线程调度。
- 对象容器与内核对象统一管理。
- 软件定时器与系统节拍。
- IPC 机制的最小实现与验证。
- Cortex-M 移植层和临界区管理。

## 推荐结构

```text
GS_Thread/
├── kernel/             # 内核对象、线程、调度、IPC、定时器
├── port/               # Cortex-M 移植层
├── bsp/                # 板级启动和基础外设
├── examples/           # 最小验证示例
└── docs/               # 设计说明和调试记录
```

## 使用建议

1. 如需继续开发，优先切到 `main`。
2. 如需引用稳定版本，可使用 `master` 的固定提交。
3. 如发现 `master` 与 `main` 偏差，应明确记录差异原因。
4. 临时实验不要直接提交到 `master`，应使用独立 feature/debug 分支。

## 质量标准

| 检查项 | 要求 |
| --- | --- |
| 稳定性 | 保持可编译、可运行、可复现 |
| 同步性 | 与 `main` 的关键结构差异可解释 |
| 可读性 | 内核关键路径不被业务代码污染 |
| 文档 | 重大差异和版本状态需要记录 |
| 风险控制 | 不接收未经验证的大范围重构 |

## 分支关系

| 分支 | 用途 |
| --- | --- |
| `main` | 推荐开发入口 |
| `master` | 历史兼容和稳定视图 |
| `release` | 里程碑与发布沉淀 |

## 维护边界

该分支不建议继续作为实验场。后续新能力应先在 `main` 或独立分支完成验证，再按需要同步到 `master`。
